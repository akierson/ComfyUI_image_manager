# ComfyUI Image Manager — Domain Glossary

## Core Concepts

**Managed Image**
An image tracked by the lineage system. Has a UUID assigned at save time, a sidecar JSON file, and a record in the SQLite database. All images saved via `ManagedSaveImage` or imported via the web UI are managed images.

**Root Image**
The first image in a refinement chain — a managed image with no parent. Can be generated via `ManagedSaveImage` (with no `ManagedLoadImage` in the workflow) or imported externally via the web UI or startup scan.

**Lineage Chain**
The ordered sequence of managed images from a root image through all its refinements. A chain can branch — one image can have multiple children if it was refined in different directions.

**Parent Image**
The managed image that was loaded as input when a child image was generated. Parentage is established automatically: if the current workflow contains a `ManagedLoadImage` node referencing a managed image, the saved output is a child of that image.

**Child Image**
A managed image whose parent is another managed image. Recorded at save time by `ManagedSaveImage` by inspecting the current workflow for a `ManagedLoadImage` node.

**Generation**
A single refinement step — the act of running a workflow that takes a parent image as input and produces one or more child images. When a workflow produces a batch of N images, all N are siblings of the same parent, not a chain.

**Root Name**
The folder name used to group all images in a lineage chain. For root images, derived by resolving all `%date:...%` tokens in the `filename_prefix` at save time (e.g. `portrait_2026-06-04`). For child images, inherited unchanged from the parent's record. Images with the same resolved prefix saved on the same day intentionally share a root name folder.

**Sidecar JSON**
A `.json` file stored beside each managed image (same name, `.json` extension). Contains: `uuid`, `parent_uuid` (null for roots), `root_uuid`, `root_name`, `created_at` (ISO 8601), and `filename` (relative path from managed root). The authoritative portable record of an image's identity and lineage. The SQLite database is derived from sidecar files and can be fully rebuilt from them.

**Managed Folder**
A single filesystem folder registered with ComfyUI's `folder_paths` system at startup under the key `managed_images`. Path is configured via the `ImageManager.managed_folder` setting in the ComfyUI settings panel; defaults to `<ComfyUI_root>/managed_output/`. Structured as `YYYY-MM-DD/<root_name>/` inside. Changing the path requires a server restart.

**Image UUID**
A UUID (version 4) generated at save time and stored in both the sidecar JSON and the PNG metadata (`lineage_id` text chunk). Stable across file renames and moves. The primary key in the SQLite database and the link between parent and child.

**SQLite Database**
A single database file in the custom node folder. Stores all managed image records indexed by UUID for fast lineage queries. Derived from sidecar JSON files — can be rebuilt by scanning the managed folder.

**ManagedSaveImage**
A custom ComfyUI node that saves images to the managed folder. On save: assigns a UUID, writes the sidecar JSON, updates the SQLite database, and detects the parent by inspecting the current workflow for a `ManagedLoadImage` node. Structures output into `YYYY-MM-DD/<root_image_name>/`.

**ManagedLoadImage**
A custom ComfyUI node that loads a managed image from a registered managed folder. Its presence in a workflow is the signal that the next `ManagedSaveImage` output is a child of the loaded image.

**Send to Comfy**
A web UI action available on any managed image. Extracts the full original workflow from the image's PNG metadata, replaces the original image source with a `ManagedLoadImage` node pointing to the selected image, embeds the selected image's UUID as `parent_id`, and loads the reconstructed workflow into the ComfyUI canvas via the frontend JS API. Does not execute — opens for editing.

**Import**
The act of bringing an external image (not generated by `ManagedSaveImage`) into the lineage system. Performed via the web UI drag-and-drop or file picker, or automatically during startup scan. Assigns a UUID, writes a sidecar JSON, and creates a DB entry. An imported image becomes a root image.

**Folder Import**
A mass import operation that moves an entire external folder of images into `managed_root`. The user provides a filesystem path; the server walks it, moves each image file into `managed_root`, writes sidecars, and upserts DB records. Structure mapping: a top-level subfolder name becomes the `root_name`; images at the folder root use their filename stem. Folders already matching the `YYYY-MM-DD/root_name/` convention have their original date and root_name preserved. Duplicate filenames (already present in DB) are skipped silently. The operation is destructive — source files are moved, not copied.

**Lineage Link**
A manual assignment of a parent-child relationship between two existing managed images, performed via the web UI. The first selected image becomes the child; the second becomes the parent. Reparenting cascades: the child and all its descendants inherit the new parent's `root_uuid` and `root_name`. Any managed image can be reparented regardless of whether it already has a parent; reassigning a child with an existing parent requires confirmation. Cycles are not permitted.

**Import Source Folder**
A filesystem path provided by the user as the source for a Folder Import. Treated as read-only-intent: files are moved out of it, but the system does not write sidecar files or modify the source in any other way. Must not overlap with `managed_root`.

**Startup Scan**
A two-phase process that runs when ComfyUI starts. Phase 1: scans the managed folder for PNGs that have no sidecar JSON and writes one in-place (sidecar-only — the PNG file is not modified). Phase 2: wipes the SQLite database and rebuilds it from all sidecar JSONs found in the managed folder. Ensures images dropped directly into the filesystem are not invisible to the system.

**Image Embedding**
A fixed-length float32 vector representing a managed image's visual content. Computed by the Embedding Backend and cached in the SQLite `images` table (`embedding BLOB`, `embedding_backend TEXT`). On cluster request, only images with no cached embedding or a mismatched backend are recomputed; the rest are loaded from the DB. Embeddings are invalidated when the active backend changes.

**Embedding Backend**
The module responsible for computing Image Embeddings. Uses `open_clip` or `transformers` (CLIP vision encoder) if either is installed in the Python environment; falls back to an RGB color histogram if neither is available. Cluster quality silently varies depending on which backend is active.

**Visual Cluster**
A group of managed images identified as visually similar by k-means over Image Embeddings. The number of clusters is chosen automatically (elbow/silhouette method — no user input required). Visual Clusters are ephemeral: computed on demand when Cluster View is activated and discarded when toggled off. They are never written to the sidecar JSON or SQLite database and carry no lineage meaning.

**Chain Tip**
The most recent image in a lineage chain — the image with the highest `created_at` value sharing a given `root_uuid`. If a root image has no descendants, it is its own chain tip. The chain tip is shown as the primary card face in the default grid view and as the leftmost (largest) thumbnail in Filmstrip View.

**Filmstrip View**
A view mode toggled from the toolbar that displays each lineage chain as a horizontally scrollable row, with images ordered newest-first (left to right). The chain tip is rendered slightly larger with a visible border; all other thumbnails are uniform. Each row is labelled with the root name and total image count. Date grouping and the folder sidebar filter apply in filmstrip view the same as in the default grid. Clicking any thumbnail opens the lightbox for that specific image.

**Cluster View**
A UI mode toggled by the user that replaces the date-grouped image grid with a cluster-grouped grid. Each group header shows the cluster index and image count. While clustering runs the grid shows a progress bar fed by server-sent events. Toggling off restores the date-grouped view. Cluster View is intended as a discovery aid: the user inspects groups, then uses Link mode to assign parent-child relationships informed by the visual groupings.

**Folder Sidebar**
A collapsible left sidebar in the web UI that displays the folder tree derived from the managed folder's on-disk structure: date (`YYYY-MM-DD`) as top-level nodes, root name as child nodes, each showing an image count. Clicking a node filters the main grid to root images in that folder. No folder selected means all images are shown. Selecting an empty folder shows an empty state. A "hide dates" toggle flattens the sidebar to root names only and simultaneously removes date group headers from the main grid.
